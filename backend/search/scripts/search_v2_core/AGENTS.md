# Search V2 Agent System

This directory contains the core logic for the "Agentic" search system (Search V2). The system is designed as a modular pipeline where specialized "agents" (Python modules) perform specific tasks to answer user queries about FDA drug labels.

The system is orchestrated by a **Controller** (`controller.py`) which manages the flow of execution based on a shared **State** object.

## Agent Descriptions

The agents are located in the `agents/` subdirectory. Here is a description of each agent and its responsibility:

### 1. Planner (`agents/planner.py`)
*   **Role:** The "brain" of the operation. It is the first agent to run.
*   **Function:**
    *   Analyzes the user's natural language query and conversation history.
    *   Determines the **Intent** (e.g., `search`, `aggregate`, `qa`, `chitchat`, `clarification`).
    *   Constructs a **Retrieval Plan**, which includes:
        *   Which SQL template to use (e.g., `search_by_set_id`, `metadata_search`).
        *   Filters to apply (e.g., specific sections, labeling types).
        *   Search terms and key entities (Brand Name, NDC, SPL ID).
*   **Output:** Updates `state.intent` and `state.retrieval["plan"]`.

### 2. DB Executor (`agents/db_executor.py`)
*   **Role:** The primary search engine.
*   **Function:**
    *   Takes the plan generated by the Planner.
    *   Selects the appropriate SQL template from `sql.py`.
    *   Constructs and executes the SQL query against the Oracle FDALabel database.
    *   Retrieves a list of matching drug labels (metadata).
*   **Output:** Populates `state.retrieval["results"]` with the raw database rows.

### 3. Postprocess (`agents/postprocess.py`)
*   **Role:** The evaluator and traffic controller.
*   **Function:**
    *   Checks the results from `db_executor`.
    *   **Fallback Logic:** If no results are found for a brand name search, it can modify the plan to retry searching by "Active Ingredient" and route back to `db_executor`.
    *   **Decision Making:** Decides if the system needs to fetch full text content ("evidence") or if the metadata is sufficient.
        *   If the user asks a complex question (e.g., "What are the adverse reactions?"), it flags `needs_evidence = True`.
        *   If the user just wants a list of drugs, it skips evidence fetching.
*   **Output:** Sets the next step to either `evidence_fetcher` or `answer_composer`.

### 4. Evidence Fetcher (`agents/evidence_fetcher.py`)
*   **Role:** The researcher.
*   **Function:**
    *   Runs only if `needs_evidence` is True.
    *   Takes the top results (drug labels) found by `db_executor`.
    *   Queries the database for specific **Label Sections** (e.g., "34084-4" for Adverse Reactions) based on the user's intent.
    *   **Text Windowing:** Extracts relevant text snippets from the label XML/text, focusing on keywords from the user's query to avoid loading massive amounts of irrelevant text.
*   **Output:** Populates `state.evidence["snippets"]` with the specific text passages needed to answer the question.

### 5. Aggregate Executor (`agents/aggregate_executor.py`)
*   **Role:** The statistician.
*   **Function:**
    *   Runs instead of `db_executor` + `evidence_fetcher` when the intent is `aggregate` (e.g., "How many labels mention headache?").
    *   Executes specialized aggregation queries to get counts, top companies, or top generic names.
*   **Output:** Populates `state.retrieval["aggregate"]` with statistical summaries.

### 6. Answer Composer (`agents/answer_composer.py`)
*   **Role:** The writer.
*   **Function:**
    *   Synthesizes the final response to the user.
    *   Combines the User Query, Metadata Results, and Fetched Evidence (text snippets).
    *   Uses an LLM to generate a natural language answer that cites the evidence (e.g., "Ozempic causes nausea [[1]]...").
    *   Also handles simple "Chitchat" or "Clarification" requests directly.
*   **Output:** Sets `state.answer["response_text"]`.

### 7. Reasoning Generator (`agents/reasoning_generator.py`)
*   **Role:** The explainer.
*   **Function:**
    *   Runs at the very end of the process.
    *   Looks at the entire execution history (`trace_log`).
    *   Generates a user-friendly summary of *how* the system solved the problem (e.g., "I searched for 'Advil', found 10 labels, and checked the 'Indications' section...").
    *   This output is displayed in the "Reasoning" panel on the frontend.
*   **Output:** Sets `state.reasoning`.

## Agent Relationships & Workflow

The **Controller** (`controller.py`) acts as a state machine, moving the system from one agent to the next based on flags set in the `state`.

**Typical Search Flow:**
1.  **Planner** determines the user wants to search for a drug. -> Next: `db_executor`
2.  **DB Executor** runs the search and finds 5 matching labels. -> Next: `postprocess`
3.  **Postprocess** sees results and determines the user asked a specific question about "Side Effects". -> Next: `evidence_fetcher`
4.  **Evidence Fetcher** downloads the "Adverse Reactions" section for those 5 labels. -> Next: `answer_composer`
5.  **Answer Composer** writes the final answer using the downloaded text. -> Next: `reasoning_generator`
6.  **Reasoning Generator** summarizes the steps. -> **Terminate**

**Fallback Flow (No Results):**
1.  **Planner** plans a search for "FakeDrugName". -> Next: `db_executor`
2.  **DB Executor** finds 0 results. -> Next: `postprocess`
3.  **Postprocess** sees 0 results. It modifies the plan to search for "FakeDrugName" as an *ingredient* instead. -> Next: `db_executor` (Retry)
4.  **DB Executor** runs the new plan.

**Aggregate Flow:**
1.  **Planner** identifies a "How many..." question. -> Next: `aggregate_executor`
2.  **Aggregate Executor** runs count queries. -> Next: `answer_composer`
3.  **Answer Composer** formats the stats into text. -> Next: `reasoning_generator`
